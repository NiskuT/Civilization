974 /* Code generation conventions that need to be recorded somewhere.
975  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
976 
977  x86
978  ~~~
979  Generated code should be entered using a JMP instruction. On
980  entry, %ebp should point to the guest state, and %esp should be a
981  valid stack pointer. The generated code may change %eax, %ebx,
982  %ecx, %edx, %esi, %edi, all the FP registers and control state, and
983  all the XMM registers.
984 
985  On entry, the FPU control word should be set to 0x027F, and the SSE
986  control word (%mxcsr) should be set to 0x1F80. On exit, they
987  should still have those values (after masking off the lowest 6 bits
988  of %mxcsr). If they don't, there is a bug in VEX-generated code.
989 
990  Generated code returns to the scheduler using a JMP instruction, to
991  the address specified in the .dispatch field of VexTranslateArgs.
992  %eax (or %eax:%edx, if simulating a 64-bit target) will contain the
993  guest address of the next block to execute. %ebp may be changed
994  to a VEX_TRC_ value, otherwise it should be as it was at entry.
995 
996  CRITICAL ISSUES in x86 code generation. The only known critical
997  issue is that the host FPU and SSE state is not properly saved
998  across calls to helper functions. If any helper references any
999  such state, it is likely (1) to misbehave itself, since the FP
1000  stack tags will not be as expected, and (2) after returning to
1001  generated code, the generated code is likely to go wrong. This
1002  really should be fixed.
1003 
1004  amd64
1005  ~~~~~
1006  Analogous to x86.
1007 
1008  ppc32
1009  ~~~~~
1010  On entry, guest state pointer is r31. .dispatch must be NULL.
1011  Control is returned with a branch to the link register. Generated
1012  code will not change lr. At return, r3 holds the next guest addr
1013  (or r3:r4 ?). r31 may be may be changed to a VEX_TRC_ value,
1014  otherwise it should be as it was at entry.
1015 
1016  ppc64
1017  ~~~~~
1018  Same as ppc32.
1019 
1020  arm32
1021  ~~~~~
1022  r8 is GSP.
1023 
1024  arm64
1025  ~~~~~
1026  r21 is GSP.
1027 
1028  ALL GUEST ARCHITECTURES
1029  ~~~~~~~~~~~~~~~~~~~~~~~
1030  The guest state must contain two pseudo-registers, guest_CMSTART
1031  and guest_CMLEN. These are used to specify guest address ranges,
1032  either of code to be invalidated, when used in conjunction with
1033  Ijk_InvalICache, or of d-cache ranges to be flushed, when used in
1034  conjunction with Ijk_FlushDCache. In such cases, the two _CM
1035  pseudo-regs should be filled in by the IR, and then an exit with
1036  one of the two abovementioned Ijk_ kinds should happen, so that the
1037  dispatcher can action them. Both pseudo-regs must have size equal
1038  to the guest word size.
1039 
1040  The architecture must a third pseudo-register, guest_NRADDR, also
1041  guest-word-sized. This is used to record the unredirected guest
1042  address at the start of a translation whose start has been
1043  redirected. By reading this pseudo-register shortly afterwards,
1044  the translation can find out what the corresponding no-redirection
1045  address was. Note, this is only set for wrap-style redirects, not
1046  for replace-style ones.
1047 */
